## 第二回評価（2026-03-22）: トッププレイヤーの思考反映度 — C+

全ファイル（約2,165行）を精読した上での厳密な評価。
設計思想「Mortalを最大限信頼し、順位戦略でのみ介入」は正しいが、介入ロジックの麻雀的精度に多数の問題がある。

---

### 1. 致命的問題

#### 1-1. アガリの無条件強制（strategy_engine.py:103）
```python
if IDX_HORA < len(mask) and mask[IDX_HORA]:
    return IDX_HORA
```
mask に hora がある場合は無条件でアガるが、トッププレイヤーの思考と矛盾する場面がある：
- オーラス1位でアガると着順が下がるケース（アガリ点が低すぎて他家のリーチ棒回収で2着落ち等）
- ダブロンの片アガリ選択が必要な場面

#### 1-2. `mortal_action == IDX_NONE` で即リターン（strategy_engine.py:111-112）
Mortal がパスを選択した場合、戦略エンジンは一切介入しない。
しかし、Mortal が鳴きチャンスをスルーしている場面で、順位状況的には鳴くべきケースがある
（オーラス4位で速度最優先など）。パスの上書きが一切できないのは設計上の大きな穴。

#### 1-3. 点数計算が30符固定（push_fold.py, game_state.py）
`estimate_hand_value` と `min_han_for_points` が30符基準でハードコード。
ピンフツモ（20符）、七対子（25符）、副露手の40符/50符など、符計算の差は打点に大きく影響。
特にオーラスの逆転条件計算で「必要翻数」が1翻ずれると戦略が根本から変わる。

---

### 2. 重大な問題

#### 2-1. 受入枚数が未使用
`game_state.py` に `estimate_acceptance_count()` が実装済みだが、`push_fold.py` から一度も呼ばれていない。
トッププレイヤーは「テンパイかどうか」だけでなく「何面待ちか」「有効牌が何枚残っているか」で攻守を決める。
良形テンパイ（両面8枚）と悪形テンパイ（ペンチャン4枚）で同じ判断は明らかに誤り。

#### 2-2. ダマテン判定の不完全さ（placement_strategy.py:323-384）
- 待ちの良さを考慮していない（悪形ダマは出アガリ率が低い）
- 巡目を考慮していない（序盤ダマと終盤ダマでは意味が違う）
- 他家の捨て牌から待ち牌が出やすいかの判断がない
- 「跳満以上なら常にダマ」（L379-382）は過度に単純化。跳満でもリーチで三倍満になれば逆転できるオーラス4位のケースを無視

#### 2-3. 副露判断が Q値差分のみ
鳴き判断が「Mortal の Q値」と「none の Q値」の比較のみ。トッププレイヤーが考慮する：
- 鳴いた後の手牌構成（守備力の変化）
- 鳴きによる打点変化（メンゼン維持のリーチ+裏ドラ vs 副露の速度）
- 他家への情報開示

これらが一切なく、Q値の閾値調整だけでは副露の質を上げられない。

#### 2-4. 安全牌選択の問題
- 現物（完全安全牌）の最優先が明示的に保証されていない
- `safety_weight` 上限0.80で、FOLD時でもQ値が20%反映される（ベタオリ時は安全度100%にすべき）
- ワンチャンス評価が弱すぎる（+0.08）

#### 2-5. 脅威度推定の限界
- 副露の打点推定がない（白ポン+中ポンと白ポンのみで同じ扱い）
- 手出し牌の内容を見ていない（中張牌の手出しと端牌の手出しでは意味が違う）
- 巡目ごとの門前テンパイ確率を考慮していない

---

### 3. 中程度の問題

#### 3-1. オーラス戦略の粒度不足
- 着順落ちリスクの計算が不十分
- 直撃条件の計算がない（「下家から直撃ロンすれば逆転」は頻出パターン）
- ノーテン罰符の考慮がない

#### 3-2. 染め手検出の粗さ（game_state.py:138-161）
- 門前ホンイツ/チンイツ（リーチホンイツなど）を全く検出できない
- 河の偏りが50%以上という閾値が甘い

#### 3-3. 東風戦非対応
`is_all_last` が「南4局」固定（game_state.py:342）。東風戦では「東4局」がオーラス。

#### 3-4. Q値比較のバグ（strategy_engine.py:153-157）
```python
if discard_q >= riichi_q * 0.90:
```
Q値が負の場合にこの比較は意味をなさない（負×0.9 は元より大きくなる）。差分ベースに変更すべき。

#### 3-5. 3人麻雀への非対応
GameState は4人固定。mortal3p と組み合わせた akagi_supreme3p が存在しない。

---

### 4. 良い点

- 設計思想「Mortal信頼 + 順位戦略補完」は正しい
- SupremeEngine のラッパーパターンがクリーンで既存インターフェースを壊さない
- 手出し/ツモ切り追跡が実装されており方向性は正しい
- オーラスの逆転条件計算（翻数別ロン/ツモ）の基本構造がある
- EVALUATION.md に修正履歴があり自己改善の意識がある

---

### 5. 結論

akagi_supreme は正しいアーキテクチャを持つが、介入ロジックの麻雀的精度が不足。現状は：
- **介入なし**の場面では Mortal と同等（=問題なし）
- **介入あり**の場面では不正確な判断で Mortal より悪くなる可能性がある

トッププレイヤーの思考を正しく反映するには、上記の致命的・重大な問題の修正が必須。
特に「受入枚数の活用」「符計算の導入」「アガリ/パスの例外処理」「待ち形評価」が最優先課題。

## 第三回評価（2026-03-22）: トッププレイヤー思考の反映度 — C+（据え置き）

全ファイル（bot.py, supreme_engine.py, strategy_engine.py, game_state.py, push_fold.py, placement_strategy.py, model.py — 計約2,200行）を精読。
第二回評価の指摘事項への修正が入っているが、根本的な設計限界と新たな問題が浮上。
**雀魂トッププレイヤー（魂天・雀聖上位）の思考を「正しく反映できているか」**の観点で厳しめに評価する。

---

### 評価基準

雀魂トッププレイヤーの判断の特徴：
1. **局収支ではなく半荘収支**で判断する（順位点の影響が極めて大きい）
2. **テンパイからはほぼ降りない**が、「何枚待ちか」「何点か」で微調整する
3. **副露判断は打点・速度・守備力のバランス**で行い、Q値のような単一指標では決まらない
4. **ベタオリは完全ベタオリ**。中途半端な「80%安全」はトッププレイヤーの思考にない
5. **手出し/ツモ切りの読みは常に行う**が、それ以上に「何が切られていないか」を読む
6. **オーラスは全く別のゲーム**。着順条件から逆算して必要打点を決め、手組み自体を変える

---

### 1. 致命的問題（Mortalより悪くなる可能性が高い箇所）

#### 1-1. アガリの無条件強制が依然として残っている（strategy_engine.py:106-107）

```python
if IDX_HORA < len(mask) and mask[IDX_HORA]:
    return IDX_HORA
```

第二回で指摘済みだが未修正。Mortalの判断（`mortal_action`）より前にmaskチェックでhoraを強制している。

**トッププレイヤーが見送るケース：**
- オーラス1位で1000点ロン → アガると供託棒を回収した2位に抜かれる（他家リーチ棒1本で逆転）
- 役満テンパイ中にチーテンのロン牌が出た場合（極端だがゼロではない）
- アガるとオーラス親番が流れ、連荘できなくなるケースでのツモ見逃し

Mortal自体がこれらのケースで`IDX_HORA`をmaskに入れつつ`IDX_NONE`を選ぶ可能性があるため、Mortalの判断を上書きして強制アガリにすることは**Mortalより確実に悪くなる**。
コメントに「the extremely rare exception (yakuman pursuit) is not worth the complexity」とあるが、オーラスの着順絡みは全く稀ではない。

**深刻度: 高**
**修正案:** Mortalが`IDX_NONE`を選んだ場合はそれを尊重する。maskにhoraがあってもMortalがhoraを選ばなかった理由がある。

#### 1-2. push_fold の safety_weight がMortalのQ値選択を大幅に歪める

strategy_engine.py の `_adjust_discard` が Q値と安全度の加重平均で牌を選び直す仕組みだが、**MortalのQ値はすでに安全度を内包している**。

Mortalのニューラルネットは訓練時に「放銃した場合の順位低下」を含めたQ値を出力している。つまりMortalが「5mを切る」と判断した場合、それは5mの攻撃価値だけでなく5mの危険度もすでに考慮した上での判断。

ここに外付けの safety.py による danger 評価を加重平均すると、**安全度が二重にカウント**される。結果として：
- Mortalが攻撃的な牌を選んでいるのに、安全寄りの牌に差し替えてしまう
- 特にリャンシャンテン以下（safety_weight=0.30〜0.65）で、Mortalの打牌選択を大きく歪める

**深刻度: 高**
**トッププレイヤーとの乖離:** トッププレイヤーは「押すと決めたら押す、降りると決めたら完全に降りる」。中途半端な加重平均はどちらの判断にもならない。

#### 1-3. 受入枚数推定が不正確（game_state.py:465-509）

`estimate_acceptance_count()` は「隣接牌があれば有効牌」と判定するが、これはシャンテン数を下げる有効牌ではなく、単に手牌に関連する牌の数を数えているだけ。

例: `1m2m4m5m 1p2p4p5p 1s2s4s` （リャンシャンテン）
- この実装: 3m,3p,3s,6m,6p,5s,6s... 等を全部カウント → 30枚以上
- 正しい受入: シャンテン数を下げる牌のみ → はるかに少ない

この不正確な値が `good_shape`/`bad_shape` の判定に使われ、push_fold の分岐を狂わせる。

**深刻度: 高**
**修正案:** libriichi の PlayerState.shanten はすでに取得している。同じく libriichi から正確な受入枚数を取得すべき。自前実装は不要。

---

### 2. 重大な問題

#### 2-1. 副露判断のロジックが順位状況に対して不十分

`_adjust_meld` はFOLD時にチーを拒否、CAUTIOUS時にQ値閾値で判断するが、**攻めの副露判断**が弱い。

トッププレイヤーが副露で速度を上げる場面：
- 中盤以降のドラポン → ほぼ100%鳴く（Mortalも鳴くが、safety_weightでNONEに差し替えられる可能性）
- 役牌ポン → 1翻確定で手を進める
- オーラスで必要打点に届く副露

`_check_pass_override` でオーラスのパス上書きを試みているが、「オーラス4位 + meld_multiplier >= 1.1」の条件が厳しすぎる。実際にはオーラス4位なら**ほとんどの副露チャンスを活かすべき**。Q値差0.15以内という制限も不要で、トッププレイヤーなら「この鳴きで手が進むかどうか」だけで判断する。

#### 2-2. ダマテン判断の待ち牌山残り評価が欠如

`should_damaten()` は待ち形の良し悪し（acceptance_count）を見るようになったが、**山に残っている枚数**を見ていない。

例: 良形リャンメン待ちでも、待ち牌が全て見えていれば出アガリしかない → リーチの威嚇価値が上がる
例: 悪形ペンチャンでも、待ち牌が3枚山にあれば十分 → ダマでもツモれる可能性

トッププレイヤーは河と自分の手牌から待ち牌の残り枚数を推定してダマ/リーチを決める。`unseen_count()` が game_state.py に実装済みなのに使われていない。

#### 2-3. 安全牌選択の精度不足

`_find_genbutsu_discard()` は現物を見つけるが、**現物が複数ある場合の優先順位**が不十分。リーチ者の河にある牌の中から選ぶ際、トッププレイヤーは：

1. 複数のリーチ者に共通する現物を最優先
2. 次に、自分の手牌進行に最も不要な現物を選ぶ
3. 後の巡目のために「使える安全牌」を温存する

現在の実装は「最も多くのリーチ者に安全な牌」を選ぶが、手牌進行への影響を見ていない。ベタオリ中でも「回し打ち」（安全牌を切りながら手を進める）はトッププレイヤーの基本技術。

#### 2-4. 脅威度推定が副露手の打点を推定していない

`apparent_threat_level()` は副露の種類（ポン/チー）とパターン（ホンイツ/トイトイ）を見るが、**推定打点**を出していない。

トッププレイヤーの判断：
- 白ポン のみ → 1翻1000〜2000点、大した脅威ではない
- 白ポン + 中ポン + ホンイツ模様 → 跳満以上確定、最大級の脅威
- ドラポン → それだけで3900点以上、もう一つ役があれば満貫

現在の実装は全てを同じ「threat += 0.6」で扱っており、打点差を反映できていない。push_fold の `estimate_risk_of_deal_in` も `base_risk = 5200` という固定値がベースで、実際の副露手の打点推定に基づいていない。

#### 2-5. 東場の戦略がほぼ空（placement_strategy.py:343-356）

`_east_strategy` は親番なら微攻撃（riichi_multiplier=1.05）、それ以外は標準。しかしトッププレイヤーは東場でも：
- 東1局で大きく負けた場合の立て直し戦略
- 東3局/東4局の点差に応じた微調整
- 3着目・4着目の親番での連荘志向

を考慮する。東場≠何も考えない、ではない。ただし、これはMortalに任せるべき範囲でもあり、介入しないという判断自体は妥当。問題は「東場だから介入しない」のではなく「Mortalが十分対応できるから介入しない」という明確な設計意図が必要。

### 3. 構造的・設計上の問題

#### 3-1. MortalのQ値の意味を正しく理解していない設計

本botの最大の構造的問題は、**MortalのQ値が何を表しているかを正しく理解した上で介入していない**こと。

MortalのQ値は「この行動を取った場合の期待順位（のスケール変換）」。つまり：
- すでに順位を考慮済み
- すでに安全度を考慮済み
- すでに打点と速度のバランスを考慮済み

これに対して外部から safety_weight や placement_adjustment を掛けることは、**Mortalがすでに考慮した要素を二重に考慮する**ことになる。

正しいアプローチは以下のどちらか：
1. **Mortalが明確に苦手とする特定のパターンのみを検出して上書き**する（例: 特定のオーラスパターンでの過小/過大評価）
2. **Mortalの出力を使わず、独自のルールベースで完全に判断する**（例: ベタオリ時は完全に safety.py ベース）

現在の「Q値と安全度の加重平均」は中途半端で、どちらのアプローチでもない。

#### 3-2. テストがゼロ

約2,200行のロジックに対してユニットテストが一つもない。特に：
- `_calculate_points()` の点数表が正しいか検証不可
- `estimate_acceptance_count()` の出力が妥当か検証不可
- オーラスの各分岐が意図通り動くか検証不可
- push_fold の各閾値が想定シナリオで正しく動くか検証不可

麻雀は場合分けが膨大なため、テストなしでは信頼性を担保できない。

#### 3-3. 3人麻雀（サンマ）非対応

GameStateは4人固定。雀魂はサンマも人気モードだが、akagi_supremeはmortal3pとの連携がない。auto_switch_model でmortal/mortal3pを切り替える仕組みがcontroller.pyにあるが、akagi_supremeはmortalのみ使用。

## 第三回評価に基づく修正（2026-03-22）

### 致命的問題の修正

#### 1. アガリ強制の削除（strategy_engine.py）
- **修正前**: `mask[IDX_HORA]` が true なら無条件でアガリを強制
- **修正後**: Mortal が `IDX_HORA` を選んだ場合のみアガリ。Mortal がパスした場合はその判断を尊重
- **理由**: オーラスの着順条件でアガリ見送りが正解のケースがある（例: 安手ロンで供託棒回収した2位に抜かれる）

#### 2. safety_weight 加重平均の廃止 → 二値（三値）判断へ（push_fold.py, strategy_engine.py）
- **修正前**: `safety_weight` (0.0〜1.0) で Q値と安全度を連続的にブレンド → 安全度の二重カウント
- **修正後**: Decision を `PUSH / MAWASHI / FOLD` の三値に変更
  - `PUSH`: Mortal のアクションをそのまま返す（Q値を一切歪めない）
  - `MAWASHI`: 安全牌を優先しつつ、同程度の安全度の中から Q値で選択（回し打ち）
  - `FOLD`: 完全ベタオリ。現物 → スジ → 壁 → ワンチャンスの純粋な安全度順
- **MAWASHI の実装**: 牌を安全度で3段階（SAFE/MODERATE/DANGEROUS）に分類し、最も安全なカテゴリの中からQ値最高の牌を選択。中間カテゴリでは安全寄り半分の中からQ値選択。
- **理由**: トッププレイヤーは「押す/降りる」の二値で考える。中途半端な混合はどちらの判断にもならない。

#### 3. 受入枚数推定の改善（game_state.py）
- **修正前**: 手牌に隣接する牌を全てカウント → 過大推定（30枚以上）
- **修正後**: `_estimate_deficiency()` を実装し、実際にシャンテン数を下げる牌のみをカウント
  - 各牌について「ツモって1枚切る」シミュレーションを行い、手牌構造が改善するかチェック
  - `_estimate_deficiency()` は面子＋部分ブロック構造をグリーディに評価
  - libriichi に受入枚数 API がないため自前実装だが、精度は大幅向上
- **テスト追加**: 完成形(deficiency=0)、テンパイ(≤1)、バラバラ(≥4) の検証

### 重大な問題の修正

### 1. 発見されたバグ（実動作に影響）

#### 1-1. 面子抽出のグリーディ順序問題（game_state.py: `_count_mentsu_and_partial`）
**問題**: 順子を先に抽出する一つの順序のみ試行。`1m1m1m2m3m4m` の場合:
- 順子優先: 1m-2m-3mを取る → 残り1m1m4m → 面子1個
- 刻子優先: 1m1m1mを取る → 残り2m3m4m → 面子2個
受入枚数推定 (`_estimate_deficiency`) がこの値に依存するため、攻守判断に直接影響。

**修正**: 順子優先・刻子優先の両方を試行し、最良の結果を採用。

#### 1-2. 現物判定の赤牌正規化漏れ（strategy_engine.py: `_find_genbutsu_discard`）
**問題**: `ACTION_TILE_NAMES[34]` = "5mr" だが、genbutsu_tiles は `tile_base()` で正規化済み（"5m"）。
赤5を手牌に持っている場合、genbutsu として認識されずベタオリで安全牌を切れない。

**修正**: 現物検索時に `tile_base()` で正規化してからルックアップ。

#### 1-3. 最終打牌プレイヤーの誤判定（strategy_engine.py: `_get_last_opponent_discard`）
**問題**: `len(p.river)` を「最近の捨て牌」の代理指標に使用。しかし河が一番長いプレイヤー ≠ 最後に切ったプレイヤー。
ポンの対象牌を誤判定し、役牌ポン/ドラポンの自動受理が不正確になる。

**修正**: `GameState._last_discard_tile` / `_last_discard_actor` で正確に追跡。

### 2. 戦略的問題（トッププレイヤー思考との乖離）

#### 2-1. オーラス1位の攻守判断が一律MAWASHI化
**問題**: `adjust_for_placement` がオーラス1位のPUSHを無条件でMAWASHIに変換。
しかし薄リード（差< 4000）では:
- ノーテン罰符(-3000)で逆転される危険
- テンパイをPUSHして早めに和了するのが防御的に正しい
トッププレイヤーは「薄リードのオーラス1位で降りてノーテン → 逆転される」を最も嫌う。

**修正**: diff_to_below < 4000 の場合は元の判断を維持（PUSHのまま）。

#### 2-2. オーラス4位が差12000以上で介入なし
**問題**: `adjust_for_placement` がオーラス4位で `diff_above > 12000` の場合、元の判断を変更しない。
しかしオーラス4位は**最悪の着順が確定**しているため、常に攻めるべき（降りても4位は変わらない）。

**修正**: FOLDを常にMawashi/PUSHに昇格。差が大きくてもPUSHよりは低いが、FOLDは許容しない。

#### 2-3. パス上書き（_check_pass_override）の適用範囲が狭い
**問題**: オーラス3位の副露上書きが「4位が近い場合」のみ。「2位に近い場合」は未対応。
また、オーラス2位で1位に僅差の場合の副露上書きもなし。
トッププレイヤーはオーラスで着順を上げるための副露を積極的に活用する。

**修正**: オーラス3位（2位に近い場合）、オーラス2位（1位に僅差の場合）の副露上書きを追加。

#### 2-4. リーチ後の他家打牌が現物として未追跡
**問題**: リーチ者の河にある牌のみを現物とする。しかし、リーチ後に他家が切った牌をロンしなかった場合、
その牌もリーチ者に対して完全安全（手牌がロックされているため）。
トッププレイヤーのベタオリではこの「後付け現物」を最大限活用する。

**修正**: `PlayerInfo.post_riichi_safe` を追加し、`_handle_dahai` でリーチ者がパスした牌を記録。
`_find_genbutsu_discard` で河の現物に加えてこのセットも参照。

#### 2-5. 断么九（タンヤオ）の打点評価漏れ
**問題**: `estimate_hand_value` が断么九を検出しない。2-8のみの手牌で1翻分の打点が欠落。
テンパイ時の攻守判断やオーラスの逆転条件計算に影響。

**修正**: 手牌に1/9/字牌がないことを検出し、+1翻を追加。

### 1. 致命的バグ: `riichi_turn` が全体巡目を格納（per-player巡目と混同）

#### 問題
`_handle_reach` が `self.turn`（全体巡目カウンタ: 0〜70）を `riichi_turn` に格納。
しかし下流の比較は全て自家巡目（0〜17）基準で行われている:

- `apparent_threat_level`: `if self.riichi_turn <= 6:` → 全体巡目6は最初の1巡目相当。事実上「早いリーチ」がほぼ検出されない
- `push_fold.py`: `if p.riichi_turn <= 6:` → 同上
- `safety.py`: `early_dealer_riichi_boost_at = 8` → 全体巡目8は2巡目。通常の早いリーチ（自家6巡目=全体巡目24）は検出不能

**影響**: 早いリーチに対する脅威度上昇、放銃リスク増加、安全牌選択の補正が全て機能しない。
トッププレイヤーが「6巡目以内のリーチは良形・高打点の可能性が高い」と判断する重要な読みが欠落。

**修正**: `riichi_turn = self.turn // 4` に変更。`_threat_of` も `self.my_turn` を渡すよう変更。

#### 2. `_threat_of` が全体巡目を `apparent_threat_level` に渡す

`_threat_of` が `self.turn`（全体巡目）を `current_turn` として渡す。
`apparent_threat_level` 内の `if current_turn >= 10:` は自家巡目10（中終盤）を意図しているが、
全体巡目10は2〜3巡目に相当し、序盤から「中終盤の読み」が発動してしまう。

**修正**: `self.my_turn`（= `self.turn // 4`）を渡すよう変更。

### 2. ダマテンパイの放銃リスクが実質無効

#### 問題
`estimate_risk_of_deal_in()` のダマテンパイ検出:
```python
base_risk = max(base_risk, 5200)
```
`base_risk` の初期値が `5200.0` なので、`max(5200, 5200) = 5200` で何も変わらない。
ツモ切り連続→手出しのダマテンパイシグナルを検出しても、リスク評価に全く反映されない。

**修正**: `max(base_risk, 7700)` に変更（ダマテンパイは満貫以上の可能性が高い）。

### 3. オーラス4位のパス上書きが保守的すぎる

#### 問題
`_check_pass_override` のオーラス4位条件: `meld_q >= none_q - 0.25`
オーラス4位は既に最悪の着順。降りても4位のまま。トッププレイヤーは4位確定なら
Q値に関係なくほぼ全ての副露を取る。閾値が保守的すぎて、十分な副露機会を活かせない。

**修正**: 閾値を `-0.45` に緩和。

### 4. ダマテン判断で山残り枚数（unseen_count）が未使用

#### 問題
`should_damaten()` が山残りを評価していない。`unseen_count()` が実装済みなのに未参照。

トッププレイヤーの判断:
- 待ち牌が1〜2枚しか残っていない → ツモ期待薄 → リーチで威嚇して相手を降ろす
- 待ち牌が3〜4枚残っている → ダマでもツモ可能 → 手を変える柔軟性を維持

**修正**: `acceptance_count <= 2` の極薄待ちではダマテンを拒否（リーチ優先）。

### 5. イーシャンテン終盤で高脅威時のFOLD判定が欠如

#### 問題
`evaluate_push_fold` のイーシャンテン終盤: `threat >= 2.0` でも MAWASHI 止まり。
しかし `threat >= 3.0`（ダブルリーチ等）で安手の場合、トッププレイヤーはベタオリする。
終盤のイーシャンテン安手で無理に攻めて放銃するのは最も避けるべきパターン。

**修正**: `threat >= 3.0 and hand_value < risk * 0.4` の条件で FOLD に遷移。
